## P1 — High (protocol correctness / data integrity)

### NEW-97: `on_start_logout()` silently drops Logout request when session is not `LoggedOn`

**Category:** Bug

**TF**: `crates/truefix-session/src/state.rs` `on_start_logout()` (~L1521-1528).

```rust
fn on_start_logout(&mut self) -> Vec<Action> {
    if self.state == SessionState::LoggedOn {
        // ... send Logout, transition to AwaitingLogout
        vec![self.send_stored(lo)]
    } else {
        Vec::new()  // ← silently dropped
    }
}
```

`on_start_logout` (the handler for `Event::StartLogout`, called when the application requests a
graceful logout) only sends a `Logout` message when `state == LoggedOn`. In any other state
(`AwaitingLogon`, `AwaitingLogout`, `Disconnected`), it returns `Vec::new()` — the application's
logout directive is silently discarded with no action and no notification.

The most impactful case is `AwaitingLogon` for an **initiator**: if the application calls
`logout()` immediately after `Engine::start` (before the counterparty's Logon has arrived), the
Logout is never sent. The initiator remains in `AwaitingLogon` until `logon_timeout` expires,
then is forcibly disconnected — the counterparty never receives a Logout.

**QFJ**: `Session.java`'s `logout()` (line ~940) unconditionally calls `generateLogout()` and
transitions state, regardless of the current session state. Even in `LOGON_RECEIVED` state, QFJ
queues the Logout to be sent once the Logon round-trip completes.

**Impact**: An initiator that decides to cancel a connection during the Logon handshake cannot do
so gracefully — the counterparty sees an abrupt TCP close (after `logon_timeout`) instead of a
proper Logout. For a multi-session engine where the application may shut down individual sessions
dynamically, this is a real operational issue.

**Fix direction**: Either send a Logout in `AwaitingLogon` state (it will arrive after the
counterparty's Logon, which is acceptable), or queue one to be sent once `LoggedOn` is reached, or
at minimum immediately transition to `Disconnected` + `Action::Disconnect` rather than no-op'ing.

### NEW-98: Dictionary-validation rejection of an out-of-order Logon does not advance `next_in_seq`

**Category:** Bug

**TF**: `crates/truefix-session/src/state.rs` `on_logon()` dictionary-validation branch (~L1344-1366).

```rust
if let Some(reject) = self.validate_app(&msg) {
    let mut actions = vec![reject];
    if logon_seq_early.is_some_and(|s| s == self.next_in_seq) {
        self.next_in_seq = self.next_in_seq.saturating_add(1);
    }
    // ... disconnect_on_error path ...
    return actions;
}
```

When a Logon fails dictionary validation and its `MsgSeqNum` equals `next_in_seq`, the sequence is
advanced. But when the Logon's `MsgSeqNum` is **greater** than `next_in_seq` (out-of-order), the
`if s == self.next_in_seq` condition fails and `next_in_seq` is **not** advanced.

The Logon is correctly rejected (Reject + optional Logout + Disconnect), but the counterparty's
outbound sequence has advanced — they sent `MsgSeqNum=N` and will next send `N+1`. TrueFix still
expects `next_in_seq` (the old value), so the next connection's first message at `N+1` looks too
high, triggering a `ResendRequest` for a gap that doesn't exist.

**QFJ**: `nextLogon()` (`Session.java` ~L1310) always advances the target sequence on a received
Logon, even when rejecting it — the counterparty's sequence number was consumed.

**Impact**: After rejecting an out-of-order Logon, the next reconnect's first message is
unexpectedly too-high, causing a spurious `ResendRequest` and potential confusion. The desync
self-heals once the resend completes, but it's unnecessary traffic and a real protocol-correctness
gap.

**Fix direction**: In the dictionary-validation rejection branch, advance `next_in_seq` whenever the
Logon's sequence is `>= self.next_in_seq` (not just `==`), or simply advance unconditionally for
any Logon that arrives with a valid `MsgSeqNum`.

### NEW-99: `SequenceReset(35=4)` bypasses `validate_app()` dictionary validation entirely

**Category:** Bug

**TF**: `crates/truefix-session/src/state.rs` `on_received()` (~L748).

```rust
if mt.as_deref() == Some("A") {
    return self.on_logon(msg);
}
if mt.as_deref() == Some("4") {
    return self.on_sequence_reset(&msg);  // ← routed before validate_app
}
// ... later: validate_app() runs for everything else
```

`on_received` routes `MsgType=4` (SequenceReset) directly to `on_sequence_reset()` before the
normal `seq.cmp(&self.next_in_seq)` dispatch — which is where `validate_app()` is called (inside
the `Ordering::Equal` branch at ~L788). Every other message type (application messages, and now
even admin messages per BUG-89/FR-011) goes through dictionary validation, but `SequenceReset`
completely skips it.

A malformed `SequenceReset` with illegal field values (e.g. a non-integer `NewSeqNo`, an unknown
tag, a missing required field in a FIXT 1.1 transport dictionary) is accepted and processed
without any dictionary-level check.

**QFJ**: `Session.java`'s `nextSequenceReset()` (line ~1539) calls `verify(sequenceReset, ...)`
which runs the full dictionary validation path (`DataDictionary.validate()`) before processing the
message — `SequenceReset` is not exempt from field-level validation in QFJ.

**Impact**: A counterparty sending a `SequenceReset` with a non-numeric or malformed `NewSeqNo`
tag has it silently accepted by the `.filter(|&n| n > 0)` / `as_int()` chain — which maps it to
`None` and then silently skips the reset — instead of being rejected as a format error. More
broadly, any dictionary-level constraint (required fields, enum values, field types) is bypassed
for this one message type.

**Distinction from NEW-84**: NEW-84 (in `005.md`) is about `SequenceReset` bypassing the
**too-high sequence-number check** (`checkTooHigh`). This item is about bypassing **dictionary
validation** (`validate_app`) — a completely different check that runs at a different layer. Both
gaps exist independently.

**Fix direction**: Move the `MsgType=4` routing to after the `validate_app()` call in the
`Ordering::Equal` branch, or add an explicit `validate_app()` call inside `on_sequence_reset()`
before processing the message.

### NEW-100: `send_stored` persists teardown Logout messages — stale Logout can be resent on next connection

**Category:** Bug

**TF**: `crates/truefix-session/src/state.rs` `send_stored()` (~L508-523), called from
`on_received()`'s latency-failure path (~L708) and identity-failure path (~L719).

When a non-Logon message fails the latency check or CompID check, `on_received` sends a `Logout`
message via `send_stored(admin::logout(...))`. `send_stored` — its name notwithstanding — stores the
message into `self.store` (keyed by `MsgSeqNum`) whenever `config.persist_messages` is true:

```rust
fn send_stored(&mut self, mut msg: Message) -> Action {
    // ...
    if self.config.persist_messages
        && let Some(seq) = msg.header.get(MSG_SEQ_NUM) ... {
        self.store.insert(seq as u64, msg.clone());  // ← Logout stored
    }
    Action::Send(msg)
}
```

If `ResetOnDisconnect` and `ResetOnLogout` are both `false`, the store is not cleared on teardown.
On the next connection, if the counterparty sends a `ResendRequest` covering this sequence number,
the stored Logout is resent as part of gap-fill recovery — the counterparty receives a spurious,
out-of-context `Logout(35=5)` message that was originally sent to signal a latency/identity failure
on a prior connection.

This is distinct from any Logout sent during the normal Logout handshake (`on_logout_msg` /
`on_start_logout`), which is also stored but is at least contextually correct to resend (it represents
a legitimate part of the session history). The latency/identity teardown Logout is a one-shot
diagnostic message that should never be replayed.

**QFJ**: QFJ's `Session.java` does not store teardown/error messages in the message store —
`disconnect()` / `generateLogout()` are sent via `sendRaw()` which bypasses storage, while
`send()` (which stores) is used only for normal sequenced traffic.

**Impact**: A counterparty performing gap recovery after a reconnect may receive an unexpected
Logout, potentially triggering their own logout/disconnect logic and tearing down the freshly
re-established session.

**Fix direction**: Use `send_raw()` (which does not store) instead of `send_stored()` for
teardown-error Logout messages, matching QFJ's `sendRaw()` usage for the same purpose. Alternatively,
remove Logout messages from the store during `enter_disconnected()` when the teardown was
error-driven rather than a normal Logout handshake.

### NEW-101: `as_int`/`as_decimal` use `.trim()` — silently accept whitespace-padded wire values

**Category:** Bug

**TF**: `crates/truefix-core/src/field.rs` `as_int()` (~L120), `as_decimal()` (~L128).

```rust
pub fn as_int(&self) -> Result<i64, ...> {
    let s = self.as_str()?;
    s.trim().parse::<i64>().map_err(...)
}

pub fn as_decimal(&self) -> Result<Decimal, ...> {
    let s = self.as_str()?;
    s.trim().parse::<Decimal>().map_err(...)
}
```

Both `as_int()` and `as_decimal()` call `.trim()` on the string value before parsing. A wire value
like `"  123  "` (with leading/trailing spaces or other whitespace) is silently accepted and parsed
as `123`, even though the FIX specification requires integer and decimal field values to be
strictly numeric with no surrounding whitespace.

The raw bytes are preserved for round-trip fidelity (the field stores the original value), but the
typed accessor returns a false-positive `Ok(123)` for what is actually a malformed wire value. This
infects `value_ok()` in the dictionary validator, which calls `as_int().is_ok()` — a
whitespace-padded integer passes type validation when it should be rejected.

**QFJ**: QFJ's `IntConverter` (`IntConverter.java`) and `DecimalConverter` call
`Long.parseLong(s)` / `new BigDecimal(s)` directly, with **no trimming** — surrounding whitespace
causes a `NumberFormatException`. The strict behavior is intentional: wire values should not
contain extraneous whitespace, and accepting them silently can mask encoding bugs.

**Impact**: A counterparty sending `"  123  "` as an integer field value passes TrueFix's type
validation but would be rejected by QFJ's strict converter — interoperability divergence. The
whitespace also round-trips to the wire on re-encode, propagating the malformed value.

**Fix direction**: Remove `.trim()` from `as_int()` and `as_decimal()`, relying on `.parse()` to
reject whitespace-padded values. If there's a concern about real-world tolerance, make strict
parsing the default and add a separate lenient accessor if needed.

### NEW-109: TLS client trust store silently falls back to *empty* — every TLS handshake without an explicit `SocketTrustStore` fails

**Category:** Bug (folded in from a later, sequential single-reviewer spot-check pass; an
in-progress/incomplete fix rather than a newly-introduced regression)

**TF**: `crates/truefix-transport/src/tls_config.rs::build_client_config` (~L219-234):

```rust
pub fn build_client_config(spec: &TlsSpec) -> Result<Arc<ClientConfig>, TlsConfigError> {
    let roots = trust_store(spec)?.unwrap_or_else(RootCertStore::empty);
    ...
```

`trust_store()` returns `None` when neither `SocketTrustStore` nor `SocketTrustStoreBytes` is
configured. In that case, `build_client_config` builds a `ClientConfig` with a completely **empty**
`RootCertStore` — zero trust anchors. Every server certificate, including one issued by a public,
universally-trusted CA, then fails webpki chain verification with `UnknownIssuer`. A TLS initiator
with `SocketUseSSL=Y` and no explicit trust store cannot complete a handshake to *any*
counterparty. This is reachable from real `Engine::start` code paths (`crates/truefix/src/lib.rs`
lines ~641, 670, 699 all call `build_client_config` directly off `.cfg`-resolved `TlsSpec`).

This is exactly the defect `specs/009-audit-005-remediation/research.md` §R2 (`NEW-11`) already
describes and plans to fix — the working tree at review time had added the `rustls-native-certs`
dependency (`Cargo.toml`/`crates/truefix-transport/Cargo.toml`) but **the actual wiring into
`tls_config.rs` had not landed yet**: `native_certs` was not imported or called anywhere in the
crate. This is a status observation on in-progress work, not an undiscovered bug in the sense of
the other items in this document — flagging so it isn't lost.

**QFJ/QFGo**: An unconfigured JVM `SSLContext` (QFJ) falls back to the JRE's OS-integrated default
trust store, not an empty one; matching that (rather than a bundled CA list) is why
`rustls-native-certs` — not `webpki-roots` — was chosen as the fallback (research.md §R2).

**Impact**: Currently blocking — any `.cfg` with `SocketUseSSL=Y` and no `SocketTrustStore`/
`SocketTrustStoreBytes` set (the common case for connecting to a counterparty whose cert chains to
a public CA) fails every connection attempt.

**Fix direction**: In `trust_store()`'s `None` branch (or `build_client_config` directly), load
`rustls_native_certs::load_native_certs()` into the `RootCertStore` instead of defaulting to
`RootCertStore::empty()`.

**Correction to `005.md`'s own record — `NEW-54` is already fixed, not open.** `005.md` (§NEW-54)

## P2 — Medium (default divergence / feature gap)

### NEW-102: `heartbeat_timeout_multiplier` default 2.0 diverges from QFJ's 1.4

**Category:** Default-Divergence

**TF**: `crates/truefix-session/src/config.rs:201` — `heartbeat_timeout_multiplier: 2.0`.

The heartbeat timeout is computed as `HeartBtInt × multiplier + 2` ticks. With the default
`HeartBtInt=30` and `multiplier=2.0`, TrueFix waits 62 seconds before disconnecting a silent peer.

**QFJ**: `DefaultSessionFactory.java` defaults `HeartBtIntTimeoutMultiplier` to `1.4`
(`Session.java` ~L210: `DEFAULT_HEARTBT_TIMEOUT_MULTIPLIER = 1.4`). With the same HeartBtInt=30,
QFJ waits 44 seconds.

**Impact**: TrueFix detects a dead peer ~40% slower than QFJ at default settings. Not a correctness
issue (an operator who sets the value explicitly sees no divergence), but a migration surprise.

**Fix direction**: Change the default from `2.0` to `1.4`, matching QFJ.

### NEW-103: `test_request_delay_multiplier` default 1.0 diverges from QFJ's ~0.5

**Category:** Default-Divergence

**TF**: `crates/truefix-session/src/config.rs:201` — `test_request_delay_multiplier: 1.0`.

The TestRequest probe fires after `HeartBtInt × multiplier` idle ticks. With the default
`HeartBtInt=30` and `multiplier=1.0`, TrueFix sends a TestRequest after 30 seconds of silence.

**QFJ**: QFJ sends a TestRequest after approximately `HeartBtInt / 2` seconds of silence
(`Session.java` ~L2035: the test-request counter threshold is `heartbeatInterval / 2`, rounded).
With HeartBtInt=30, QFJ probes after ~15 seconds.

**Impact**: TrueFix takes twice as long as QFJ to probe a silent peer at default settings, delaying
detection of a half-open connection.

**Fix direction**: Change the default from `1.0` to `0.5`, matching QFJ's behavior.

### NEW-104: `ValidateFieldsOutOfOrder` config default `false` diverges from QFJ's `true`

**Category:** Default-Divergence

**TF**: `crates/truefix-config/src/builder.rs:625` —
`validate_fields_out_of_order: bool_key(map, "ValidateFieldsOutOfOrder", false)`.

When a `.cfg` does not set `ValidateFieldsOutOfOrder`, TrueFix defaults to `false` — out-of-order
fields (e.g. a body field appearing in the header, or a trailer field before the body) are silently
accepted. The decoder does set `msg.fields_out_of_order = true` (used for reporting), but
validation does not reject the message.

**QFJ**: `DataDictionary.java` defaults `checkFieldsOutOfOrder` to `true`
(`DataDictionary.java` ~L95: `setCheckFieldsOutOfOrder(true)` in the constructor). An out-of-order
field produces a `SessionRejectReason=14` (Tag specified out of order).

**Impact**: A message with fields in the wrong section (header/body/trailer) passes TrueFix
validation but would be rejected by QFJ — interoperability divergence. This is especially relevant
for fields that the static `is_header`/`is_trailer` classification misroutes (since those checks
also use the same classification).

**Fix direction**: Change the default in `builder.rs` from `false` to `true`, matching QFJ.

### NEW-105: No `CancelOrdersOnDisconnect` support

**Category:** Feature-Gap

**TF**: `crates/truefix-session/src/state.rs` — no `cancel_orders_on_disconnect` field or handling.

QFJ's `CancelOrdersOnDisconnect` config key (Session setting) causes the session to automatically
send `OrderCancel` (`35=F`) requests for all open orders when the TCP connection is lost, before
the session disconnects. This is a safety mechanism for trading systems: when connectivity to an
exchange is lost, outstanding orders remain live on the exchange side, and the firm may want to
cancel them automatically rather than risk unmanaged exposure.

TrueFix has no equivalent: on disconnect, the session simply transitions to `Disconnected` and
drops. The `Application` trait's `on_disconnect()` callback is the only notification, but the
application must implement all cancel logic itself, including order tracking, message
construction, and sequencing — with no session-level support for sending messages during teardown.

**QFJ**: `Session.java`'s `disconnect()` (~L1620) checks `cancelOrdersOnDisconnect` and calls
`generateOrderCancelRequests()` before the socket is closed, sending cancels for all tracked open
orders using the session's own sequence-number space.

**Impact**: Trading system operators migrating from QFJ must implement their own cancel-on-disconnect
logic at the application layer, which is error-prone (the session is tearing down, sequencing is
complex, and there may not be a reliable send path during teardown).

**Fix direction**: Add a `cancel_orders_on_disconnect: bool` config field, and in
`enter_disconnected()` (or just before it), if set, emit `Action::Send` for order-cancel messages
for all tracked orders. This requires an order-tracking mechanism, which may need to be an
`Application`-layer callback (`fn open_orders(&self) -> Vec<Order>`) or a session-level store.

### NEW-106: No `PossResend(97)` support

**Category:** Feature-Gap

**TF**: `crates/truefix-session/src/state.rs` — no handling of tag 97 (`PossResend`).

FIX protocol defines `PossResend(97)` as a boolean field on application messages indicating "this
message may have been previously sent" — distinct from `PossDupFlag(43)`, which is a session-level
tag indicating the message is a duplicate at the transport layer. `PossResend` is set by the
**application** (not the session) when it re-sends a business-level message that may not have been
confirmed, and the receiving application is expected to perform its own deduplication.

QFJ passes `PossResend` to the `Application::fromApp()` callback so the application can check
whether it has already processed the message and take appropriate action (skip, deduplicate, or
process as new).

TrueFix does not parse, surface, or pass `PossResend` to the application at all. An inbound message
carrying tag 97 is decoded normally; the field lands in `message.body` (or `message.header`,
depending on tag classification) and is never inspected by the session layer. The
`Application::from_app()` callback has no `poss_resend: bool` parameter.

**QFJ**: `Session.java`'s `nextAppMessage()` (~L2480) extracts `PossResend(97)` and passes it to
`Application.fromApp(message, sessionId, possResend)` — the application receives a boolean flag and
can deduplicate accordingly.

**Impact**: An application built on TrueFix cannot distinguish a first-time message from a
possible re-send, preventing correct application-level deduplication. For trading systems where
idempotency is critical (e.g. avoiding double execution of an order), this is a real gap.

**Distinction from 005.md's missing-features table**: The missing-features table lists 6 items
(`EncryptMethod`, `Username`/`Password`, `LastMsgSeqNumProcessed` validation, `MaxMessagesInResendRequest`,
`MessageStore::refresh()`, `FixtDictionaries` validate/dispatch helper). `PossResend` is **not**
among them.

**Fix direction**: Parse `PossResend(97)` from the inbound message in `on_received()` and pass it
to `Application::from_app()` (or `from_admin()`) as an additional parameter, matching QFJ's
signature. Alternatively, leave it in the `Message` and document that the application should check
`msg.body.get(97)` itself — but this is less ergonomic and easy to miss.

## P3 — Low (hardening / edge case)

### NEW-107: `frame_length` does not verify SOH byte at the CheckSum trailer's end

**Category:** Bug

**TF**: `crates/truefix-core/src/framing.rs` `frame_length()` (~L74).

BUG-46/FR-047 (feature 007) added a verification that `buf[total-7..total-4] == b"10="` — confirming
the CheckSum tag is present at the expected position. However, the function does **not** verify that
`buf[total-1] == SOH` (the byte immediately after the three-digit checksum). QFJ's
`FIXMessageDecoder` verifies the complete `10=\\d{3}\\x01` trailer pattern.

A malformed frame where `BodyLength` is correct, `10=` appears at the right position, and the
checksum digits are numeric, but the byte after the checksum is not SOH (e.g. it's a letter or
digit), would be accepted as a complete frame by `frame_length`. The subsequent `tokenize_validated`
call would then parse the non-SOH byte as the start of the next field, potentially misframing the
remainder of the stream.

**Impact**: Very narrow — in practice, FIX encoders always emit `10=XXX<SOH>` as the final 7 bytes,
so a real message always has SOH at this position. The gap only matters for adversarial or
deeply-malformed input that happens to have a correct `BodyLength` and `10=` prefix but a wrong
trailing byte.

**QFJ**: `FIXMessageDecoder` (line ~120) checks `header[headerLength-1] == 0x01` as part of its
trailer validation.

**Fix direction**: Add `buf.get(total - 1) == Some(&SOH)` to the existing `10=` check.

### NEW-108: No file log or message store rotation / size limits

**Category:** Perf/Hygiene

**TF**: `crates/truefix-log/src/file.rs` `FileLog::open()`;
`crates/truefix-store/src/file.rs` `BodyLog`.

`FileLog` opens the log file in append-only mode and writes indefinitely — there is no rotation,
truncation, or size cap. A long-running session accumulates every log entry in a single file that
grows without bound.

Similarly, `BodyLog` (the file-based message store) appends every sent message body to a single
file with no rotation or compaction. A session that sends millions of messages over its lifetime
accumulates a body file that can grow to many gigabytes, with `read()` (used during resend)
performing `seek` + `read` for each message — O(1) per message, but the file handle and disk
footprint grow unbounded.

**QFJ**: QFJ's `FileLog` and `FileStore` also have no built-in rotation — this is parity, not a
divergence. QFJ deployments typically use external log rotation (e.g. `logrotate`) and rely on
session resets (which truncate the body file) to bound growth. However, QFJ at least documents this
as an operational concern; TrueFix does not.

**Impact**: Disk exhaustion on long-running deployments without external rotation. Not a correctness
issue — operational hygiene only.

**Fix direction**: Document the need for external log rotation in operational docs. Optionally, add
a `MaxFileLogSize` config key that triggers rotation when the log/body file exceeds a threshold,
matching QFJ's `FileLogMaxSize` (though QFJ itself does not implement this — it's a common
deployment-level concern).

### Suggested fix priority

1. **NEW-109** (TLS client trust store empty by default) — currently blocking every
   no-explicit-trust-store TLS connection; the dependency (`rustls-native-certs`) is already
   pulled in, only the `tls_config.rs` wiring is missing.
2. **NEW-100** (stale teardown Logout in store) — cheapest to fix (`send_raw` instead of
   `send_stored`), prevents a real cross-connection spurious-Logout scenario.
3. **NEW-97** (`on_start_logout` silently drops) — small fix, affects initiator shutdown.

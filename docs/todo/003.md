## P0 — Fresh protocol-correctness bugs (new findings, not in 001.md/002.md)

### BUG-05: `on_logon` never rejects a Logon whose own MsgSeqNum is too low

**TF**: `crates/truefix-session/src/state.rs:835-887` (`on_logon`). `disposition` (`logon_seq.cmp
(&self.next_in_seq)`, computed at `:841`) is consulted twice — once to advance `next_in_seq`
(`:842-844`, `Ordering::Equal` only) and once after building the Logon response
(`:872-882`, `Ordering::Equal`/`Ordering::Greater` only). The state-transition block in between
(`:850-869`, `Role::Acceptor if self.state == AwaitingLogon => { ...LoggedOn... }`) gates purely on
session `state`, never on `disposition`. **Verified directly by reading `state.rs:835-887`**: a
Logon with `MsgSeqNum < next_in_seq` and no `PossDupFlag` falls through every `disposition` match
arm as `_ => {}`, yet the state machine still transitions to `LoggedOn` and sends back its own
Logon response — full acceptance, `next_in_seq` left un-advanced and now desynced from the
consumed slot.

**QFJ**: `Session.java` — `nextLogon` calls `verify(logon, false, validateSequenceNumbers)` before
any Logon-acceptance code runs; `verify` → `isTargetTooLow` → `doTargetTooLow`: without PossDup,
generates a Logout ("MsgSeqNum too low...") and aborts — no state transition, no Logon reply.
**QFGo**: `in_session.go`/`logon_state.go` route Logon through the same `verify`/`checkTargetTooLow`
gate used for every other inbound message type before accepting.

**Failure scenario**: a reconnecting counterparty (stale cache, replay, or deliberate downgrade)
sends a Logon with a too-low MsgSeqNum and no PossDup. TrueFix accepts it as fully valid instead of
rejecting with Logout+disconnect like both references — a real, externally-observable protocol
divergence, and a fresh sequence-accounting inconsistency to boot. **Zero test coverage**: no test
in `crates/truefix-session/tests/` sends a Logon below `next_in_seq`.

**Fix direction**: when `disposition == Some(Ordering::Less)` without valid PossDup justification,
route through the same `reject_logon` path the (closed) duplicate-Logon-rejection fix already uses
(`state.rs:810-817`), before the state-transition/response block.

### BUG-06: `on_sequence_reset` (plain Reset mode) accepts a decreasing `NewSeqNo` — anti-replay hole

**TF**: `crates/truefix-session/src/state.rs:784-802`. The gap-fill branch correctly guards against
rewinding (`if ns >= self.next_in_seq`, `:794`); the non-gap-fill (`else`) branch has **no such
guard** — `self.next_in_seq = ns;` unconditionally (`:798`), confirmed by direct read.

**QFJ**: `Session.java` (`nextSequenceReset`) explicitly three-way branches: `newSequence >
expected` → accept; `newSequence < expected` → log + `generateReject(..., VALUE_IS_INCORRECT,
NewSeqNo.FIELD)`, never applied. **QFGo**: `in_session.go` (`handleSequenceReset`) — identical
three-way switch, decreasing case rejected via `doReject(msg, valueIsIncorrectNoTag())`.

**Failure scenario**: `SequenceReset` is admin-typed and (per `on_received`'s dispatch at
`state.rs:541-543`) routed directly to `on_sequence_reset`, **entirely bypassing** the PossDup
anti-replay checks (BUG-08/GAP-08, closed) that protect every other inbound message type. An
attacker or malfunctioning counterparty sending `SequenceReset(NewSeqNo=1, GapFillFlag=N/absent)`
rewinds `next_in_seq` backward, re-opening the window to replay/re-accept previously-processed
sequence numbers as "new" — a genuine anti-replay hole, not present in either reference engine.
**Zero test coverage**: `crates/truefix-session/tests/recovery.rs:312-319`
(`sequence_reset_reset_mode_sets_expected`) only tests an *increasing* `NewSeqNo`.

**Fix direction**: in the `else` branch, only apply `next_in_seq = ns` when `ns > next_in_seq`;
otherwise emit a session Reject (`SessionRejectReason::ValueIsIncorrect`, ref tag 36).

### BUG-07: acceptor routing ignores SubID/LocationID/SessionQualifier — GAP-47-configured sessions become permanently unroutable

**This is the cross-cutting interaction the third-party review specifically asked to be
double-checked** — confirmed real, and newly introduced by feature 005 itself (harmless before
005, because the fields it now populates didn't exist on the `.cfg` path until GAP-47 closed).

**TF**: `crates/truefix-transport/src/lib.rs:1430-1433` (`route_and_run`) builds its session
lookup key via `SessionId::new(begin, their_target, their_sender)` — the 3-arg constructor, which
hardcodes `sender_sub_id`/`sender_location_id`/`target_sub_id`/`target_location_id`/
`session_qualifier` to `None` (`crates/truefix-session/src/session_id.rs:29-49`, `new`). But
`AcceptorBuilder::with_session` registers sessions keyed by `config.session_id()`
(`crates/truefix-session/src/config.rs`), which uses `SessionId::new_full` (`session_id.rs:51-70`)
and **does** populate all 8 fields — including the 5 GAP-47 (005) just wired `.cfg` parsing for
(`crates/truefix-config/src/builder.rs:444-448`). `SessionId` derives `PartialEq, Eq, Hash` on the
whole struct, so an exact match on every field is required to route a connection to its session.

**Verified**: `route_and_run`'s key-building line (`lib.rs:1433`) reads
`SessionId::new(begin.clone(), their_target.clone(), their_sender.clone())` exactly as reported —
confirmed by direct read during synthesis, not just trusted from the sub-audit.

**Failure scenario**: two acceptor `[SESSION]` blocks sharing one `SocketAcceptPort`, same
BeginString/SenderCompID/TargetCompID, distinguished only by `SessionQualifier=BOOK1`/`BOOK2` —
exactly the scenario `002.md`'s own GAP-47 closure note calls out as newly possible. Both register
under their real, qualifier-populated `SessionId`; every inbound Logon builds a qualifier-`None`
lookup key that matches **neither** — the connection always falls through to "no session found"
regardless of which counterparty sent it. **Zero test coverage**: no test in
`crates/truefix-transport/tests/` drives a live connection against a `SessionQualifier`- or
SubID-distinguished acceptor session (`crates/truefix-config/tests/session_identity_mapping.rs`
only tests `.cfg` → `SessionConfig` resolution, never live routing).

**Fix direction**: `route_and_run` needs to extract tags 50/142/57/143 (SenderSubID/
SenderLocationID/TargetSubID/TargetLocationID) from the inbound Logon and build the lookup key via
`new_full`. `SessionQualifier` has no wire tag (matches QFJ) — a qualifier-disambiguated session
needs either a dedicated per-qualifier template mechanism or an explicit documented fallback; this
sub-part needs a design decision, not just a mechanical fix.

### BUG-08: store-write errors are silently swallowed everywhere in transport — zero operator-visible signal on persistence failure

**TF**: `crates/truefix-transport/src/lib.rs` — confirmed by direct grep during synthesis:
`let _ = store.save_and_advance_sender(...)` (`:1229`), `let _ =
store.set_next_sender_seq(...)`/`set_next_target_seq(...)` (`:1061-1062`, `:1249`), `let _ =
store.reset(...)` (4 call sites: `:659,760,1160,1698`), and `if let Ok(stored) =
store.get(1, out - 1).await` (`:646`, silently skips resend re-hydration on error). No
`log.on_event(...)` (or any log call) anywhere near these sites — total silence on failure.

**QFJ** wraps every store call and reports failures, e.g. `Session.java` (`getExpectedSenderNum`)
catches `IOException` and calls `getLog().onErrorEvent("getNextSenderMsgSeqNum failed: " +
e.getMessage())`.

**Failure scenario**: a Postgres-backed `SqlStore` loses its connection mid-session (network blip,
DB restart). `persist_sent` keeps returning from `save_and_advance_sender` with the error dropped;
the session keeps sending application messages believing they're durably persisted for resend,
while none actually are — no log line, no metric, nothing to alert an operator until a
`ResendRequest` arrives and the store silently returns an empty/wrong range. This directly
undermines the resend-durability guarantee GAP-38/39's persistence work exists to provide.

**Fix direction**: route every currently-swallowed store `Result::Err` through
`services.log.on_event(...)` at minimum (already available today, no trait change needed) —
pairing with a future `Log::on_error_event` (GAP-40) is a nice-to-have, not a blocker for this fix.

### BUG-09: spurious full store reset (sequence numbers + message history wiped) on every restart that lands inside an already-active schedule window

**TF**: `crates/truefix-transport/src/lib.rs:1687-1729` (`run_scheduled_initiator`).
`was_in_session` is initialized to `false` unconditionally on every process start (`:1689`),
regardless of whether the store's persisted state already belongs to the current window. On the
first loop iteration, `decide_schedule_action(&schedule, false, now)`
(`crates/truefix-session/src/schedule_reset.rs`) returns `Enter` whenever `now` is already inside
the schedule, triggering `store.reset()` (`lib.rs:1698`) — deleting all stored messages and
resetting both sequence numbers to 1, **every time the process restarts mid-window**, even for a
routine crash-and-restart with no legitimate reason to reset.

**Root cause**: GAP-38 (005, closed) added `MessageStore::creation_time()` and persists it
correctly in all 5 backends — but **nothing reads it back** (see GAP-48 below). QFJ's
`resetIfSessionNotCurrent()` (`Session.java`, invoked from the constructor and from Logon handling)
only resets when the persisted creation time falls *outside* the current schedule window
(`isCurrentSession()` / `SessionSchedule.isSameSession`), specifically to avoid this false-positive
reset. TrueFix persists the exact data needed to make this distinction and then doesn't consult it.

**Failure scenario**: any process restart (deploy, crash-restart, orchestrator rolling update) that
happens to land inside an active session-schedule window silently wipes sequence numbers and
resend history — real, unflagged data loss, not merely a missed optimization. **Zero test
coverage** for restart-mid-window (`crates/truefix-transport/tests/scheduled.rs` only covers
`connects_when_in_session`/`does_not_connect_out_of_session`).

**Fix direction**: seed `was_in_session` from whether `store.creation_time()` already falls within
the current schedule window, mirroring QFJ's `isCurrentSession`, instead of hardcoding `false`.

### GAP-48: `MessageStore::creation_time()` is write-only — persisted correctly by all 5 backends (GAP-38) but never read back by any session/transport/engine logic

Broader than BUG-09 above (which is the concrete symptom): `grep -rn "\.creation_time("
crates --include="*.rs"` finds callers only in `crates/truefix-store/tests/*.rs` — zero call
sites in `truefix-session`, `truefix-transport`, or `truefix`. `truefix-session`'s state has no
`creation_time` field/concept at all. QFJ actively consumes the persisted value beyond just the
scheduled-restart case BUG-09 covers — `Session.isCurrentSession()` more generally gates any
schedule-window comparison on it. Tracking this as its own item (distinct from BUG-09) because the
general "nothing consumes this" fact is broader than the one restart-mid-window symptom already
fixed by BUG-09's suggested direction — a full fix should make `creation_time` a first-class,
session-layer-visible concept, not just patch the one call site.

### BUG-10: `jdbc:h2:` and `jdbc:sqlserver://` don't match the real QuickFIX/J JDBC URL grammar — BUG-04 (005, "closed") is incomplete for two of its target schemes

Independently found by both the transport/config audit and the Engine-facade audit (same root
cause, two vantage points) — high-confidence finding.

**`jdbc:h2:` silently misroutes to a bogus SQLite file, no error at all (the worse of the two).**
`is_jdbc_sql_scheme()` (`crates/truefix-config/src/builder.rs:1024-1030`) recognizes `jdbc:h2:` as
SQL-family and dispatches it through `sql_store_config` with no further scheme check. But
`connect_pool` (`crates/truefix-store/src/sql.rs:360-384`) only explicitly recognizes
`postgres://`/`postgresql://`/`mysql://` — **anything else falls into an unconditional `else`
branch that treats it as a SQLite file path** (`SqliteConnectOptions::from_str` only strips a
`sqlite:`/`sqlite://` prefix if present, confirmed against vendored `sqlx-sqlite-0.9.0/src/
options/parse.rs:168-180`; otherwise the whole string is used verbatim as a filename), with
`.create_if_missing(true)` set. A `.cfg` with `JdbcURL=jdbc:h2:mem:quickfixj` (a real, common
QuickFIX/J test-fixture pattern) resolves without error and silently creates/opens a SQLite file
literally named `h2:mem:quickfixj` in the working directory, persisting real message-store data
into it — worse than the original BUG-04 (which at least failed loudly with
`UnsupportedBackend`), because here the scheme-recognition table *advertises* support that isn't
implemented and the fallback masks it entirely. **Zero test coverage**
(`crates/truefix-config/tests/store_and_log_mapping.rs` only exercises `jdbc:sqlite:`, never
`jdbc:h2:`).

**`jdbc:sqlserver://` doesn't match real Microsoft JDBC grammar.** Real MSSQL JDBC URLs use
semicolon-delimited properties with no path segment at all:
`jdbc:sqlserver://host[:port][;databaseName=X;user=Y;password=Z]`. TrueFix's
`is_jdbc_mssql_scheme` (`builder.rs:1032-1034`) strips `jdbc:` and hands the rest to
`MssqlStore::parse_url` (`crates/truefix-store/src/mssql.rs:35-67`), which **requires**
`user:password@host[:port]/database` syntax — a real QFJ-style URL
(`jdbc:sqlserver://localhost:1433;databaseName=quickfixj`) has neither `@` nor `/`, so it fails at
the first `split_once('@')` with a raw `StoreError::Backend`, not a clean `UnsupportedBackend`.
Test coverage (`store_and_log_mapping.rs:247,365`) only exercises TrueFix's own invented
path-based form, never real QuickFIX/J MSSQL `.cfg` syntax — this directly undercuts BUG-04's
stated purpose (drop-in QuickFIX/J `.cfg` compatibility) for the MSSQL branch specifically.

**Minor, same area (P2)**: `splice_credentials` (`builder.rs:1047-1060`) does no percent-encoding
of `JdbcUser`/`JdbcPassword` before splicing into the URL authority — a password containing `@`,
`:`, or `/` corrupts the resulting URL.

**Fix direction**: either implement real H2 support or remove `jdbc:h2:` from
`is_jdbc_sql_scheme` (don't advertise support that silently falls back to the wrong backend);
teach `MssqlStore`'s URL parser the real `;`-delimited grammar, keeping the current path-based form
as an additional accepted TrueFix-native shorthand.

### BUG-11: orphaned acceptor/initiator tasks on partial multi-session `Engine::start` failure — resource leak, uncontrollable running sessions

**TF**: `crates/truefix/src/lib.rs:267-568` (`Engine::start`). Iterates acceptor groups then
singles, pushing successfully-started acceptors (`JoinHandle<()>`) and initiators
(`SessionHandle`) into local `Vec`s as it goes. If a *later* session/group fails with
`continue_on_error == false`, the function does `return Err(e)` — dropping the local
`acceptors`/`initiators`/`failover_initiators` `Vec`s before `Engine { .. }` is ever constructed.
Tokio's `JoinHandle::drop` does **not** abort the task; `SessionHandle` has no `Drop` impl
(`crates/truefix-transport/src/lib.rs:336-356`) — dropping its `control` sender just flips
`control_open = false` in the running session's select loop, making the session
**uncontrollable but not disconnected** (it keeps running the FIX session indefinitely).

**Failure scenario**: a 5-session `.cfg` where sessions 1-3 start fine (acceptors + an initiator)
and session 4 fails to bind with `ContinueInitializationOnError=N` (the default) — `Engine::start`
returns `Err`, but the caller never gets a handle to the 3 sessions that already came up; they
keep listening/connected forever with no way to shut them down short of killing the process.

**Fix direction**: build the `Engine` (or at minimum abort/join everything collected so far)
before returning `Err`, or return the partial `Engine` alongside the error so the caller can shut
it down cleanly.

## P1 — Fresh bugs (real but narrower)

### BUG-12: `FieldType::value_ok` doesn't format-check `UtcTimeOnly` or `UtcDate` — malformed values silently pass dictionary validation

**TF**: `crates/truefix-dict/src/model.rs:113-141`. No match arm for `Self::UtcTimeOnly` or
`Self::UtcDate` — both fall into a catch-all `_ => true` (the comment there claims this is
intentional for "date/time-only" fields generally, but doesn't hold up). `truefix_core::Field`
already has working typed parsers for both (`as_utc_time_only`, `as_utc_date_only` in
`crates/truefix-core/src/field.rs`) — simply never invoked here.

**QFJ**: `DataDictionary.java` (`checkValidFormat`) has explicit `case UTCDATE:`/`case
UTCTIMEONLY:` validated cases. Note: QFJ's switch has **no** `case UTCDATEONLY:` either, so
TrueFix's separate non-check of `UtcDateOnly` matches upstream and should *not* be "fixed" to
diverge from QFJ — only `UtcTimeOnly`/`UtcDate` are genuine TrueFix-only regressions.

**Failure scenario**: a FIX44 field typed `UTCTIMEONLY` (e.g. tag 273 `MDEntryTime`) with a
garbled value like `"not-a-time"` passes `check_field_types` instead of being rejected
`IncorrectDataFormat`. Confirmed untested: `crates/truefix-dict/tests/field_types_extended.rs`
only asserts the *positive* case for `UtcDate`, never a negative one.

**Fix direction**: add `Self::UtcTimeOnly => field.as_utc_time_only().is_ok()` and `Self::UtcDate
=> field.as_utc_date_only().is_ok()` arms.

### BUG-13: unbounded read buffer driven by attacker-controlled `BodyLength` — memory-exhaustion DoS

**TF**: `crates/truefix-transport/src/framing.rs:13-43` (`frame_length`). Parses `BodyLength`
(tag 9) with no upper bound and computes `total = body_start + body_len + 7` with no sanity check.
The caller (`classify_buffered`/read loop, `crates/truefix-transport/src/lib.rs:819-948`) keeps
`buf.extend_from_slice`-ing every read until `buf.len() >= total`, with **no maximum buffer size
anywhere** in the read path.

**Failure scenario**: any connection (pre- or post-Logon, acceptor or initiator) sends
`8=FIX.4.4\x019=999999999\x01...` then trickles bytes slowly (or never completes), pinning an
ever-growing `Vec<u8>` for that connection's lifetime — a straightforward DoS vector, especially on
an acceptor port not gated by `AllowedRemoteAddresses`. Not previously tracked anywhere.

**Fix direction**: add a configurable (or sane hardcoded) max-body-length/max-buffer-size check in
`frame_length`, closing the connection once `body_len` exceeds it.

### BUG-14: `MssqlStore` never validates its `sessions_table`/`messages_table` identifiers before splicing into SQL — the one backend inconsistent with its siblings

**TF**: `crates/truefix-store/src/mssql.rs` — `connect_with_config` calls no
`valid_identifier`-style check before `ensure_schema()`/`get_seq`/`set_seq`/`save`/`get`/`reset`/
`creation_time`/`save_and_advance_sender`, all of which interpolate `{sessions}`/`{messages}`
directly into T-SQL strings via `format!`. Every sibling backend *does* validate:
`SqlStore` (`crates/truefix-store/src/sql.rs`), `SqlLog`/`MssqlLog`
(`crates/truefix-log/src/{sql,mssql}.rs`). Config-sourced today (not attacker-triggerable), but a
real defense-in-depth inconsistency, and produces a confusing raw `tiberius` SQL-syntax error
instead of a clean `StoreError::Backend("... is not a valid SQL table identifier")` on a typo'd
name.

**Fix direction**: port `sql.rs`'s `valid_identifier` function into `mssql.rs`, call it in
`connect_with_config`.

### BUG-15: `BodyLog::reset()` never calls `fsync`, even when `FileStoreSync=Y` — durability guarantee violated on reset

**TF**: `crates/truefix-store/src/file.rs:161-165`. `fs::write(&self.path, [])` has no accompanying
sync call, unlike `BodyLog::append()` (conditionally syncs) and `SeqFile::reset()`/`write()` (does
sync). `FileStoreSync=Y` is configured expecting durability; if the OS crashes after `reset()`'s
truncated `body` file write but before it's flushed — while the (correctly-synced) `seqnums` file
already shows `(1,1)` — recovery sees a reset sequence state with a `body` file that still contains
pre-reset messages, an inconsistent state where old messages could be replayed against post-reset
sequence numbers.

**Fix direction**: gate `fs::write`/truncation on `self.sync` with a following sync call, matching
`append()`'s discipline.

### BUG-16 (group-level `ContinueInitializationOnError`): collapses to the first grouped session's flag, not per-session

**TF**: `crates/truefix/src/lib.rs:309-311`:
```rust
let continue_on_error = members.first().is_some_and(|m| m.session.continue_initialization_on_error);
```
When two `[SESSION]` blocks share a `SocketAcceptPort` (grouped via `AcceptorBuilder`), only the
*first* member's `ContinueInitializationOnError` decides whether a bind/store/TLS failure for the
whole group is tolerated or fatal — a later session's strict preference (`N`, wants a hard failure
surfaced) is silently overridden by an earlier session's lenient one (`Y`), or vice versa.
Compounds with an already-disclosed simplification (`Services`/TLS material also taken only from
the first group member, `lib.rs:303-360`, documented in the code's own comments).

**Fix direction**: use the strictest (most conservative — i.e. `false` wins) flag across all group
members, or reject group configurations with conflicting `ContinueInitializationOnError` values at
resolve time with a clear error.

### BUG-17: AT regression floor stale after 005 added new scenarios (353 vs actual 373)

**TF**: `crates/truefix-at/tests/coverage.rs:38-44` still asserts `total_runs >= 353`. Verified by
direct instantiation of `server_suite()`: it now produces **373** scenario-runs — 005 added
`GAP08_PossDupOrigSendingTimeAfterSendingTime` and `GAP18a_DuplicateLogonRejected`, each run
across all 9 `SUITE_VERSIONS` (2 scenarios × 9 versions = 18, plus 2 more accounting for the
remaining delta to 373). The test still passes, but the floor no longer reflects actual coverage —
a future accidental drop of up to ~20 scenario-runs would go undetected.

**Fix direction**: bump the literal to `373`. Trivial, one-line fix — flagged here rather than
just fixed inline because a review of *why* it drifted (was this the last of the 005-era doc
updates, or is there a process gap in "bump the floor" being a checklist item for scenario
additions?) is worth a beat before treating it as done.

## Problems found in previously-"closed" items (002.md/005 claims that are incomplete)

These are not new gaps in the GAP-NN sense — they're re-openings of specific 005 closures that
turned out correct-as-implemented but incomplete in scope or dead in practice. Each verified by
directly reading the current implementation, not by re-reading the closure note.

- **GAP-26 (header/trailer repeating groups) — implementation correct, but entirely dormant.**
  `decode_with_groups` (`crates/truefix-core/src/codec/decode.rs:68-88`) is never called from any
  production path — the real decode path (`crates/truefix-transport/src/lib.rs:924,1420`) always
  calls the flat `decode()`. Compounding: even if wired in, `validate_groups`
  (`crates/truefix-dict/src/validate.rs:203-219`) walks `message.body.fields()`, which filters out
  `Member::Group` entries — group content would be invisible to the existing validator. And
  `truefix-core/src/tags.rs`'s hardcoded `is_header`/`is_trailer` classification still doesn't
  include `NoHops`(504)/`HopCompID`(628)/`HopSendingTime`(629)/`HopRefID`(630) — the one realistic
  FIX standard-header group — so even a direct caller would misroute it to "body." **The GAP-26
  feature doesn't do anything for any TrueFix user today.**
- **GAP-27 (per-message custom `fieldOrder`) — implementation correct, but entirely dormant.**
  `MessageDef.field_order` is parsed correctly and `encode_with_order`
  (`crates/truefix-core/src/codec/encode.rs:23-58`) applies it correctly — but nothing ever reads
  `field_order` back. Codegen's generated `encode()` (`crates/truefix-dict/src/codegen.rs:688-691`)
  is literally `self.0.encode()`, the plain no-order path. A dictionary declaring `ordered` has zero
  effect on any message TrueFix actually encodes.
- **GAP-28/GAP-32 (dictionary version metadata / BeginString-match validation) — mechanism
  correct, but orphaned from real data.** `crates/truefix-dict/src/validate.rs:33-49` is correctly
  implemented and tested, but **zero shipped `.fixdict` files** (all 8 legacy + all 9 freshly
  regenerated FIX-Repository-sourced ones from this session's dict work) declare a `version-meta`
  directive — the check never fires against any dictionary TrueFix actually ships.
- **GAP-33 (bundled dictionary coverage) — field/message counts still accurate, but the closure's
  cited mechanism no longer exists.** `002.md`'s closure text names `crates/truefix-dict/src/
  qfj_xml.rs` as the regenerating tool — that file was deleted this session (licensing correction,

  (**GAP-56**, tracked below).
- **GAP-39 (transactional save+advance) — correct for SQL/MSSQL/Redb, but `FileStore`/

  `save_and_advance_sender`; both silently fall back to the trait default (two independent,
  separately-fsync'd writes) — **the store backend requiring no external dependency, and thus the
  most likely default choice for a production deployment mirroring QFJ's own default `FileStore`,

- **BUG-04 (JdbcURL scheme recognition) — see BUG-10 above.** Correct for

## P2 — New minor bugs / tooling gaps found by this audit

- **GAP-50 / BUG-18**: `fix_repository.rs::flatten_members` (`crates/truefix-dict/src/
  fix_repository.rs:547-560`, used for header/trailer resolution) has no cycle/depth guard, unlike
  its sibling `resolve_entries` (which does, though that guard is itself effectively dead code
  since `resolve_entries` never recurses with `depth != 0`). A malformed future/different Repository
  XML edition with a self-referencing component chain would cause unbounded recursion in the CLI
  tool. Doesn't trigger against the 9 real vendored sources today (latent, not live).
- **GAP-51**: `fix_repository.rs::parse_messages`'s doc comment
  (`crates/truefix-dict/src/fix_repository.rs:302-304`) claims a synthetic `id_by_name`/`by_id`
  registration the implementation doesn't actually do. Currently harmless, but makes the invariant
  that keeps `&components_by_id[&ref_id]` (a panicking index) safe more fragile than the docstring
  implies for a future maintainer.
- **GAP-52**: `fix_repository.rs::convert`'s enum-emission is O(fields × enums) — a linear scan of
  the whole enum map per field. Build-time only (not a correctness issue), but will keep getting
  slower as more Repository editions are added; a `BTreeMap` range query would be O(log n + k).
- **BUG-19**: `SocketConnectTimeout` is applied to plain and TLS direct-connects
  (`crates/truefix-transport/src/lib.rs`, verified both `connect_initiator_with` and
  `connect_initiator_tls` pass `config.connect_timeout` through) but **not** to the proxy connect
  path — `connect_initiator_via_proxy`/`connect_initiator_via_proxy_tls` call
  `proxy::connect_through_proxy` directly, which has no timeout of its own (including the
  hand-rolled `connect_http`'s byte-at-a-time read loop). A session behind a proxy that accepts the
  TCP connection but never completes the SOCKS/CONNECT handshake hangs the initiator connect
  attempt indefinitely.
- **GAP-53**: `CipherSuites` typo silently degrades to a `CryptoProvider` with zero usable cipher
  suites instead of a clear config-time error (`crates/truefix-transport/src/tls_config.rs:161-179`
  filters by name match with no "did any name actually match" check) — surfaces as an opaque
  `rustls::Error` at handshake time instead.
- **GAP-54**: `strip_trusted_proxy_header`'s PROXY-protocol header peek loop
  (`crates/truefix-transport/src/proxy.rs:193-227`) has no overall timeout — a trusted-source
  connection that opens but sends nothing hangs that connection's task indefinitely. Low severity
  (trusted-address list is normally small/operator-controlled).
- **BUG-20**: `BodyLog::append()` has a TOCTOU race on the recorded byte offset under concurrent
  writers to the same store instance (`crates/truefix-store/src/file.rs:109-126`, offset read via
  `fs::metadata` outside the index lock). **Latent, not live** — confirmed all current store call
  sites in `truefix-transport` are sequential, un-spawned `.await`s within one task per session —
  but would corrupt the index the moment any future caller writes concurrently to one instance.
- **BUG-21**: `Engine::shutdown()` (`crates/truefix/src/lib.rs:591-598`) aborts acceptors and stops
  failover-initiators but doesn't touch plain (non-failover) `self.initiators` — likely intentional
  given the sync/async mismatch (`SessionHandle::logout()` is async, `shutdown()` is sync), but the
  doc comment doesn't disclose the asymmetry, reading as if it's a full stop.
- **GAP-55**: percent-encoding gap for `JdbcUser`/`JdbcPassword` when splicing into a `jdbc:` URL
  (see BUG-10's "minor, same area" note above).
- **BUG-22**: malformed `ResendRequest` (missing/zero `BeginSeqNo`) is silently dropped instead of
  rejected. `crates/truefix-session/src/state.rs:742-744` (`on_resend_request`): `if begin == 0 ||
  begin > end { return Vec::new(); }` — a `ResendRequest` missing tag 7, or with `BeginSeqNo=0`,
  produces no response at all, not even a session Reject; since `ResendRequest` is admin-typed it
  never reaches the dictionary-level required-field check either. QFGo:
  `in_session.go` (`handleResendRequest`) explicitly rejects with `RequiredTagMissing(tagBeginSeqNo)`/
  `RequiredTagMissing(tagEndSeqNo)` when either tag is absent, rather than silently ignoring. Low
  severity (malformed-input hygiene, not a protocol-correctness break) — distinguish "genuinely
  missing tag" (should reject) from "tag present but `begin > end`" (current silent-skip behavior,
  which is fine to keep as-is).
- **GAP-56**: stale shipped-dictionary provenance header comments — 8 `.fixdict` files still say
  "Generated by truefix-dict's QuickFIX-XML conversion tool," naming the now-deleted `qfj_xml.rs`.
  Cosmetic; a follow-up regeneration pass (using the current tooling, without changing content)
  would fix the header text.

## Known-accepted, not a gap (Rust-unsuited or already covered by a TrueFix equivalent)

Per this audit's instructions: items below were considered and explicitly ruled out — not
tracked as TODOs, listed here so they aren't re-litigated by a future pass.

- **`ClosedResendInterval`/`RejectMessageOnUnhandledException`/`MaxScheduledWriteRequests`**

  "unhandled exception" class distinct from typed `Result` errors, and no internal write queue to
  bound). Reconfirmed correct, not a gap.
- **`MessageStore::refresh()`/`close()`** (GAP-37) — QFJ's `refresh()` exists to re-sync from a
  store mutable by other threads in QFJ's concurrent session model; TrueFix's sans-IO single-owner
  architecture has no analogous staleness. Non-issue by design, not a missing feature.
- **`RedbStore::with_session_id`** — re-verified still correct (MVCC-safe, single-writer
  serialization not corruption) under the file-locking constraint originally flagged in `001.md`.
  No regression since 004.
- **QFJ's `JdbcStore` connection-retry/health-check features** vs `truefix-store/src/sql.rs` —
  sqlx's pool already does connection-liveness checking (`test_before_acquire`, on by default) with
  no `.cfg`-expressible custom-query hook to wire onto; already covered by `002.md`'s
  `JdbcConnectionTestQuery` documentation-accuracy note, no new gap.
- **`crates/truefix/examples/*.rs`** — all four cross-checked against current
  `Application`/`SessionConfig`/`AcceptorBuilder`/`BusinessReject` signatures and confirmed to
  compile clean (`cargo check -p truefix --examples`); no stale API usage found.
- **Crate-level `#![cfg_attr(not(test), deny(clippy::unwrap_used, clippy::expect_used,
  clippy::panic, clippy::indexing_slicing))]`** (`truefix-session`, `truefix-core`, `truefix-dict`
  all carry this) — confirmed by construction why no wire-triggered panics were found in `state.rs`
  or the codec layer during this audit. A real strength, not a gap; noted for completeness per
  this audit's instructions rather than filed as an item.
- **TrueFix's stricter `Country`/`Exchange` format checks** vs QFJ's unchecked `COUNTRY` type — a
  TrueFix advantage, not a gap.
- **`UtcDateOnly`'s own missing format check in `value_ok`** — matches an equivalent, apparently
  unintentional omission in QFJ's own `checkValidFormat` switch (no `case UTCDATEONLY:` there
  either). Not a TrueFix-specific divergence; don't "fix" this one in isolation without separately
  confirming QFJ's own behavior is considered a bug upstream.
- **SQL injection defense**: `SqlStore`/`SqlLog`/`MssqlLog` all validate table-name identifiers
  before `format!`-interpolation and use parameterized binds for data values — safe. (`MssqlStore`
  is the one exception, tracked as BUG-14 above, not accepted.)
- **MongoDB `(session_id, seq)` unique index** — confirmed present, idempotently created,
  `unique(true)`; `save()`'s `update_one(...).upsert(true)` is independently atomic at the document
  level regardless, so the index is defense-in-depth, not load-bearing either way.

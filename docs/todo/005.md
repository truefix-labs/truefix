## Classification — not everything below is a "bug"

This document's severity labels (P0–P3) describe *impact if it's a defect*, but not every item is a
production-code correctness defect. Reading the P0/P1/P2/P3 buckets as "N bugs of severity X" without
this table will overstate the real defect count. Each item is tagged with one of:

- **Bug** — a real production-code correctness/data-loss/security/DoS defect. This is the category
  worth prioritizing for a release.
- **Feature-Gap** — QFJ/QFGo has a capability TrueFix simply doesn't implement at all; the existing
  code isn't wrong, it just doesn't do this yet.
- **Test-Gap** — a gap in the `truefix-at` acceptance-test harness or scenario catalog (an assertion
  that's too weak, or a scenario category that's missing). The *production* session/transport code
  this would exercise may or may not itself be correct — the gap is that the test suite can't tell.
- **Tooling-Only** — affects the `generate-dict`/`generate-code` CLI or `codegen.rs`/`build.rs`
  dictionary-regeneration pipeline, not the shipped, hand-verified `.fixdict` files or any runtime
  behavior a deployed engine exhibits today. Only matters if/when a dictionary is regenerated.
  Overlaps with "Bug" for the underlying converter/codegen logic itself, but the blast radius never
  reaches a running session.
  - **Note**: `crates/truefix-dict/src/parser.rs` is *not* tooling-only — it's the runtime dictionary
    loader (used by `DataDictionary::extend()` and any caller that loads a `.fixdict` at runtime, not
    only by `build.rs`), so parser.rs findings (NEW-43/44) are tagged **Bug**, not Tooling-Only.
- **Perf/Hygiene** — behavior is correct; this is an allocation/algorithmic-complexity/style
  observation, not a defect.
- **Diagnostic-Only** — the final result/state is always correct; only logs, error messages, or
  operational visibility are degraded.
- **Default-Divergence** — TrueFix's default config value differs from QFJ/QFGo's. Not a logic error
  — an operator who sets the value explicitly sees no divergence at all.
- **Parity-Note** — TrueFix's behavior matches QFJ/QFGo's own documented/accepted behavior; flagged
  for awareness, not because it's wrong.
- **Refuted** — a candidate defect from an earlier pass that a later re-verification pass in this
  same document disproved. Kept in the document as an audit trail, not as an open item.

## P0 — Critical (protocol-correctness break, data-loss risk, or DoS vector)

### NEW-01: PossDup messages not exempt from latency check — recovery tears down session

**TF**: `crates/truefix-session/src/state.rs` `latency_ok()` (~L282-296), called from
`on_received()` (~L700) and `would_reject_before_processing()` (~L243).

`latency_ok` checks `SendingTime(52)` against the local clock with `max_latency` (default 120s)
but has **no `PossDupFlag(43)` exemption**. A resent message (PossDup=Y) carries its *original*
SendingTime; if that timestamp is older than `max_latency`, the check fails and triggers a
Logout ("SendingTime accuracy problem") + disconnect.

`poss_dup` is computed *after* `latency_ok` is called in `on_received`, so the latency gate
fires unconditionally. `would_reject_before_processing` likewise calls `latency_ok` first.

**QFJ**: `Session.java` `verify()` explicitly gates `checkLatency` on `!possDup` — a PossDup
message's original SendingTime is expected to be stale and is never a latency violation.

**Impact**: Any gap-fill recovery spanning more than `max_latency` seconds of history spuriously
tears down the session. This is the single most serious correctness defect in the current codebase.

**Fix direction**: Add a `poss_dup: bool` parameter to `latency_ok` (or check `PossDupFlag`
inside it); skip the latency check when `poss_dup == true`. Both call sites already have or can
cheaply obtain the PossDup flag.

### NEW-02: `BeginSeqNo=0` in ResendRequest silently ignored — peer's resend demand unanswered

**TF**: `crates/truefix-session/src/state.rs` `on_resend_request()` (~L935-960).

```rust
let begin = msg.body.get(BEGIN_SEQ_NO)
    .and_then(|f| f.as_int().ok())
    .filter(|&s| s > 0)   // ← filters out 0
    .map_or(0, |s| s as u64);
// ...
if begin == 0 || begin > end { return Vec::new(); }  // ← silently drops the request
```

In the FIX protocol, `BeginSeqNo=0` is a valid value meaning "from the beginning" (i.e., from
sequence number 1). QFJ treats 0 as starting from seq 1. TrueFix filters it out, maps it to 0,
then the `begin == 0` guard silently returns nothing — the peer's resend request goes unanswered.

**Note**: `EndSeqNo=0` (meaning "to infinity") is correctly handled elsewhere in the same
function. Only `BeginSeqNo=0` is broken.

**Fix direction**: Treat `BeginSeqNo=0` as 1 (the first sequence number). Remove the
`.filter(|&s| s > 0)` or special-case 0 → 1 before the `begin == 0` guard.

### NEW-03: Acceptor-side `config.reset_on_logon` not implemented

**TF**: `crates/truefix-session/src/state.rs` `on_connected()` (~L447-469),
`on_logon()` (~L1190-1206).

`on_connected` checks `config.reset_on_logon` only in the `Role::Initiator` branch; the
`Role::Acceptor` branch returns `Vec::new()`. `on_logon`'s acceptor arm uses
`inbound_reset_flag` (the peer's `ResetSeqNumFlag=141`) for the response, but never checks
`config.reset_on_logon` to proactively reset its own sequences when a Logon is received.

In QFJ, `ResetOnLogon` is primarily an **acceptor** setting: when set, the acceptor resets both
sequence numbers to 1 upon receiving a Logon, regardless of whether the initiator sent
`ResetSeqNumFlag=Y`. An acceptor configured with `ResetOnLogon=Y` in TrueFix will silently not
reset.

**Distinction from BUG-28**: BUG-28 covers the `ResetSeqNumFlag` *echo/verify* handshake
(protocol-level `141=Y` negotiation). NEW-03 is about the local `ResetOnLogon` *config* being
completely inert for the acceptor role — a different mechanism.

**Fix direction**: In `on_logon`'s acceptor branch, check `config.reset_on_logon` before
building the response; if true, call `reset_sequences(true)` and push `Action::ResetStore`.

## P1 — High (data integrity / security / feature defect)

### NEW-04: `decode`/`tokenize_validated` does not enforce `MAX_BODY_LEN`

**TF**: `crates/truefix-core/src/codec/decode.rs` `tokenize_validated()` (~L103-129).

Only `frame_length` (`framing.rs`) checks the 16 MiB `MAX_BODY_LEN` cap. A direct caller of
`Message::decode` (bypassing `frame_length`) can decode an unbounded message — a memory-exhaustion
vector. This is also the prerequisite for NEW-05 (checksum overflow in debug builds).

**Distinction from BUG-23**: BUG-23 is about `frame_length`'s integer overflow (now indirectly
mitigated by the `MAX_BODY_LEN` check in `frame_length`). NEW-04 is that the `decode` path
**entirely bypasses** `frame_length` and thus the cap.

**Fix direction**: Add `if declared_bl > MAX_BODY_LEN { return Err(DecodeError::BodyLengthTooLarge { ... }); }`
in `tokenize_validated`, after parsing `declared_bl`.

### NEW-05: `fix_repository.rs` convert() — repeating-group delimiter and member list are wrong

**TF**: `crates/truefix-dict/src/fix_repository.rs` `convert()` (~L430-445).

`resolve_entries()` returns ALL `MsgContents` entries for a repeating component, including the
count field (indent=0). The code then does:
```rust
let members: Vec<Member> = resolved.into_iter().map(|(m, _)| m).collect();  // includes count tag!
let delimiter = members.iter()
    .find_map(|m| match m { Member::Field(t) => Some(*t), _ => None })
    .unwrap_or(count_tag);  // ← count tag itself is the first Field → delimiter = count tag
```

The converter would emit `group 453 NoPartyIDs 453 453,448,447,452` (delimiter=453, count tag
in members). The correct format (confirmed in shipped `FIXLATEST.fixdict` and `FIX42.fixdict`)
is `group 453 NoPartyIDs 448 448,447,452` (delimiter=448=first body field, count tag NOT in
members). The `orchestra.rs` converter is correct (excludes count tag from members).

**Impact**: Only affects dictionaries regenerated via `generate-dict --format fix-repository`.
Shipped `.fixdict` files are hand-normalized (correct).

**Fix direction**: Exclude the count-field entry from `members`; set `delimiter` to the first
*remaining* `Member::Field`. Mirror `orchestra.rs`'s correct approach.

### NEW-06: FIX 5.0/SP1/SP2 `crack_*` dispatchers cannot distinguish versions

**TF**: `crates/truefix-dict/src/codegen.rs` `version_begin_string()` (~L480-520),
generated `crack_fix50`/`crack_fix50sp1`/`crack_fix50sp2` functions.

```rust
fn version_begin_string(name: &str) -> &'static str {
    match name {
        "FIX50" | "FIX50SP1" | "FIX50SP2" => "FIXT.1.1",
        // ...
    }
}
```

All three crack functions guard on `begin_string() != Some("FIXT.1.1")`. FIX 5.x messages all
carry `BeginString=FIXT.1.1` and are distinguished only by `ApplVerID(1128)`, which the
dispatchers ignore. A FIX 5.0 SP2 message would match `crack_fix50` and be dispatched into
FIX 5.0 structs (wrong types).

**Fix direction**: Have `crack_fix50/sp1/sp2` consult `ApplVerID(1128)` (or accept it as a
parameter) instead of relying solely on BeginString.

### NEW-07: `MultipleCharValue` enum check in `allows()` is not per-token

**TF**: `crates/truefix-dict/src/model.rs` `FieldDef::allows()` (~L155-165).

```rust
match self.field_type {
    FieldType::MultipleValueString | FieldType::MultipleStringValue => value
        .split(' ').all(|tok| self.values.iter().any(|v| v == tok)),
    _ => self.values.iter().any(|v| v == value),  // MultipleCharValue falls here
}
```

`MultipleCharValue` is NOT in the per-token split arm; it falls into `_` which does whole-string
matching. A valid wire value `"A B"` (two single chars) is incorrectly rejected. Meanwhile
`value_ok()` *does* split `MultipleCharValue` per-token (`s.split(' ').all(|tok| tok.chars().count() == 1)`)
— an inconsistency between format-checking and enum-checking.

**Fix direction**: Add `FieldType::MultipleCharValue` to the
`MultipleValueString | MultipleStringValue` match arm.

### NEW-08: `MongoStore` lacks atomic `save_and_advance_sender`

**TF**: `crates/truefix-store/src/mongo.rs`.

`SqlStore`, `MssqlStore`, `RedbStore`, and `FileStore` all override `save_and_advance_sender`
with a single transaction (GAP-39/FR-018). `MongoStore` only has `async fn save` (L165) — no
`save_and_advance_sender` override. It falls back to the trait's default two-call impl
(`save()` then `set_next_sender_seq()`), leaving the crash window the trait doc warns about.
MongoDB 4.0+ supports transactions.

**Fix direction**: Add a transactional `save_and_advance_sender` override using a MongoDB
transaction.

### NEW-09: `MssqlLog::parse_url` missing semicolon-form support and percent-decode

**TF**: `crates/truefix-log/src/mssql.rs` `parse_url()` vs
`crates/truefix-store/src/mssql.rs` `parse_url()`.

The store's `parse_url` supports `sqlserver://host;databaseName=X;user=Y;password=Z`
(BUG-10/FR-019, via `parse_semicolon_form`) and `percent_decode` on credentials
(GAP-55/FR-020). The log's `parse_url` has **neither** — it goes straight to
`split_once('@')`. A `JdbcURL` that works for the store fails to parse for the log;
credentials containing `%40`/`%3A`/`%2F` authenticate correctly in the store but fail in the log.

**Distinction from BUG-70**: BUG-70 is about `@` in passwords causing mis-split in both files.
NEW-09 is that the log's `parse_url` is an older/simpler version missing two entire features
(semicolon-form + percent-decode) that the store version has.

**Fix direction**: Port `parse_semicolon_form` and `percent_decode` from the store's `mssql.rs`
to the log's `mssql.rs`.

### NEW-10: 7 validation config keys marked `Implemented` but never wired in `builder.rs`

**TF**: `crates/truefix-config/src/keys.rs` (L60-77) vs
`crates/truefix-config/src/builder.rs` `resolve_validator()` (~L540-580).

`resolve_validator` sets only 5 toggles on `ValidationOptions`

|-----|---------------------------|---------|---------------------|
| `ValidateFieldsHaveValues` | `validate_fields_have_values` | `true` | Cannot disable empty-value check |
| `ValidateUnorderedGroupFields` | `validate_unordered_group_fields` | `true` | Cannot disable group ordering check |

### NEW-11: TLS client with no trust store configured → empty root store → all handshakes fail

**TF**: `crates/truefix-transport/src/tls_config.rs` `build_client_config()` (~L245).

```rust
trust_store(spec)?.unwrap_or_else(RootCertStore::empty)
```

When `SocketUseSSL=Y` is set without `SocketTrustStore`/`SocketTrustStoreBytes`, the root store
is empty, so rustls rejects every server certificate. QFJ falls back to the JVM's default trust
store. As written, TLS is unusable without an explicit trust store — a usability bug and parity
gap.

**Fix direction**: Use `rustls-native-certs` (or `webpki-roots`) as a default when no explicit
trust store is configured.

### NEW-12: PROXY v2 headers > 4096 bytes silently dropped, then misread as FIX

**TF**: `crates/truefix-transport/src/proxy.rs` `peek_proxy_header()` (~L200, ~L225-260).

`peek_proxy_header` peeks into a fixed 4096-byte buffer. A PROXY v2 header with TLVs exceeding
4096 bytes (spec maximum is 64 KiB) causes `parse_proxy_header` to return `None`, then
`n >= buf.len()` triggers and the function returns `peer` *without consuming any bytes*. The
remaining PROXY-protocol bytes are then misread as the FIX stream.

**Distinction from 003.md B15**: B15 reported a 256-byte buffer (raised to 4096). The 4096 cap
still cannot accommodate the full 64 KiB v2 maximum.

**Fix direction**: Dynamically grow the peek buffer (or read incrementally) until the full header
is available or the outer timeout fires.

### NEW-13: PROXY header slow-arrival gives up after ~50ms — 5s timeout budget unused

**TF**: `crates/truefix-transport/src/proxy.rs` `peek_proxy_header()` (~L240-260).

The inner loop is bounded to 10 iterations with 5ms sleeps between peeks (~50ms total). A trusted
proxy that delivers its header across several TCP segments over >50ms (but well within the 5s
outer `PROXY_HEADER_TIMEOUT`) causes the loop to return `peer` (no header) after 10 iterations,
leaving the partial header in the buffer to be misread as FIX.

**Fix direction**: Loop until the outer `tokio::time::timeout` fires, rather than a fixed
iteration count.

### NEW-14: `run_scheduled_initiator` blocks schedule-boundary and stop detection during connect

**TF**: `crates/truefix-transport/src/lib.rs` `run_scheduled_initiator` (~L2080-2200).

The loop awaits `connect_initiator_with(...).await` inline. When `config.connect_timeout` is
`None` (the default), a hanging/black-holed connect blocks the loop indefinitely — `loop_stop`
(the stop flag) and schedule-boundary transitions (`Exit`) cannot be observed until the connect
resolves. The 200ms poll cadence is effectively defeated during any connect attempt.

**Distinction from BUG-25**: BUG-25 is about the `JoinHandle` from `spawn_session_io` never being
awaited (so `current` stays `Some` forever after a drop). NEW-14 is about the connect *attempt
itself* blocking the stop/schedule-detection loop.

**Fix direction**: Wrap the connect in `tokio::select!` with the stop flag and a timeout.

### NEW-15: AT `ExpectDisconnect` conflates timeout with disconnect — false-positive risk

**TF**: `crates/truefix-at/src/runner.rs` `read_message()` (~L247-263),
`Step::ExpectDisconnect` (~L209-214).

`read_message` returns `None` for ALL of: EOF (clean disconnect), read error, AND timeout
(`Err(_)` from `timeout()`). `ExpectDisconnect` treats `None` as pass. So if a server bug causes
the session to stay connected but silent, the scenario PASSES `ExpectDisconnect` after 3s.

**Fix direction**: Distinguish EOF (real disconnect → pass) from timeout (silent → fail) in
`read_message`'s return type.

### NEW-16: AT decode failures silently swallowed — false positives and misleading errors

**TF**: `crates/truefix-at/src/runner.rs` `read_message()` (~L234).

`return decode(&raw).ok()` — a malformed-but-framed message that fails `decode` returns `None`,
identical to "no message". So a server sending a garbled message before disconnecting would make
`ExpectDisconnect` pass (false positive), and an `Expect` step would report "got nothing" rather
than "got undecodable message" (misleading).

## P0 — Critical (data-loss / complete functional failure)

### NEW-54: MongoStore `set_seq` writes to literal key `"field"` — sequence persistence completely broken

**TF**: `crates/truefix-store/src/mongo.rs` `set_seq()` (~L155).

```rust
let field = if sender { "sender" } else { "target" };
let update = doc! { "$set": { field: seq as i64 } };
```

The `bson::doc!` macro's colon syntax stringifies the identifier `field` to the literal string
`"field"` — it does **not** evaluate the variable. Every `set_next_sender_seq` /
`set_next_target_seq` call writes to a key named `"field"`, leaving `sender` / `target` untouched
at their `ensure_session_row` initial value of 1. Compare `reset()` in the same file, which
correctly uses string literals (`"sender"`, `"target"`).

**Impact**: Mongo backend sequence-number persistence is completely non-functional. After any
`set_next_sender_seq` call, `next_sender_seq()` still reads 1. Every restart begins from (1, 1).

**QFJ/QFGo**: QFGo's MongoStore correctly writes to `sender`/`target` fields.

**Fix direction**: Use the arrow form `doc! { "$set": { field => seq as i64 } }` or construct the
document programmatically with `bson::Document::insert`.

### NEW-56: `enter_disconnected` conflates `reset_on_logout` and `reset_on_disconnect`

**TF**: `crates/truefix-session/src/state.rs` `enter_disconnected()` (~L1394-1401).

```rust
if self.config.reset_on_disconnect || self.config.reset_on_logout {
    self.reset_sequences(true);
```

`enter_disconnected` is the shared teardown path for **both** graceful Logout received
(`on_logout_msg`) and TCP drop (`Event::Disconnected`). The two config flags are checked with
`||`, so each leaks into the other's scenario:
- `reset_on_logout=true` alone triggers a reset on an ungraceful TCP drop.
- `reset_on_disconnect=true` alone triggers a reset on a graceful Logout.

In QFJ, these are distinct code paths: `disconnect()` checks `resetOnDisconnect`; logout
handling checks `resetOnLogout`.

**Fix direction**: Pass a `reason` enum into `enter_disconnected` and check only the
applicable flag.

### NEW-76: `BodyLog::reset` truncates the file before acquiring the index lock

**TF**: `crates/truefix-store/src/file.rs` `BodyLog::reset()`.

truncates to 0, then append writes at EOF=0 but records offset X → orphaned/corrupt index). Not
currently triggerable (single driver task serializes), but the lock ordering is fragile.

**Fix direction**: Acquire the index lock before truncating.

### NEW-84: `SequenceReset-GapFill` bypasses too-high/too-low sequence verification entirely

**Severity: P1 (protocol correctness — can cause silent, permanent message loss)**

**TF**: `crates/truefix-session/src/state.rs`'s `on_received` routes `MsgType=4` (SequenceReset)
directly to `on_sequence_reset(&msg)` (line 748) **before** the normal `seq.cmp(&self.next_in_seq)`
dispatch (`Ordering::Equal/Greater/Less`, lines 786-866) ever runs. `on_sequence_reset`'s gap-fill
branch (lines 1177-1189) then applies `if ns >= self.next_in_seq { self.next_in_seq = ns; ... }`
unconditionally — there is no path where a gap-fill SequenceReset with a too-high `MsgSeqNum` of its
own gets queued/enqueued for later re-verification once outstanding gaps close.

**QFJ**: `Session.java`'s `nextSequenceReset` (line 1539) calls `verify(sequenceReset, isGapFill,
isGapFill)` — when `isGapFill` is true, **both** `checkTooHigh` and `checkTooLow` are active, so a
gap-fill SequenceReset whose own `MsgSeqNum` is too high is enqueued via `doTargetTooHigh` (the same
as any other too-high message) rather than applied immediately, and one that's too low invokes
`doTargetTooLow`.

**Impact**: A malformed or out-of-order gap-fill SequenceReset (its own `MsgSeqNum` higher than
expected) jumps `next_in_seq` forward immediately instead of being queued behind the real gap —
skipping verification that the reset itself arrived in the correct position in the stream. Combined
with the existing gap-fill `queue.retain(|&seq, _| seq >= ns)` (which discards any queued messages
below the new target), a spurious or reordered gap-fill message can cause **permanent, silent loss**
of legitimately queued messages that would otherwise have been drained once the real gap closed.

**Distinction from BUG-96** (`004.md`, in-progress): BUG-96 is about `drain_queue` failing to
dequeue *skipped* messages after a valid jump is accepted (already fixed in the current tree per the
`queue.retain(...)` call tagged `BUG-96/FR-027`). NEW-84 is a different, earlier problem: whether the
jump itself should ever be accepted unconditionally in the first place, before any dequeue logic
runs.

**Fix direction**: Route `SequenceReset` through the same too-high/too-low dispatch every other
message type gets (at minimum for the gap-fill case — QFJ conditions both checks on `isGapFill`),
queuing it via the normal `Ordering::Greater` path when its own sequence is ahead of expectation,
instead of processing it unconditionally the moment it's received.

### NEW-85: A `Logout` with a too-high `MsgSeqNum` is queued and answered with a `ResendRequest` instead of an immediate `Logout` reply

**Severity: P2 (protocol correctness — connection hangs instead of terminating cleanly)**

**TF**: `crates/truefix-session/src/state.rs`'s `validLogonState`-equivalent check (line 733) lets
`Some("5")` (Logout) through before logon, but *after* logon, a Logout with `MsgSeqNum >
next_in_seq` falls into the generic `Ordering::Greater` branch (lines 815-842) exactly like any
application message — it gets queued (`self.queue.insert(seq, msg)`) and a `ResendRequest` is sent
instead of being processed as a Logout.

**QFJ**: `nextLogout` (`Session.java:1457`) calls `verify(logout, false, false)` — **both**
`checkTooHigh` and `checkTooLow` are explicitly disabled for Logout, so QFJ *always* processes an
inbound Logout immediately (replies with its own Logout, honors `resetOnLogout`, then disconnects),
regardless of the Logout's own sequence position.

**Impact**: When a peer sends a Logout whose `MsgSeqNum` is ahead of what TrueFix expects (e.g. the
peer already advanced its own outbound counter for reasons TrueFix hasn't caught up on), TrueFix
queues the Logout and requests a resend instead of terminating the session — the peer, having
already sent its Logout and likely torn down its own send side, will never see a response and never
satisfy the ResendRequest, and the session hangs in `LoggedOn` state until the heartbeat-timeout
disconnect eventually fires (if `check_latency`/heartbeat checks are even active).

**Fix direction**: Special-case `MsgType=5` (Logout) the same way `MsgType=2` (ResendRequest,
BUG-29) is already special-cased in the `Ordering::Greater` branch — process it immediately (skip
the too-high check) rather than queuing it.

### NEW-86: A queued too-high `ResendRequest` is fully reprocessed (double-resent) once `drain_queue` reaches it

**Severity: P2 (protocol correctness — duplicate resend traffic on gap recovery)**

**TF**: `crates/truefix-session/src/state.rs`'s `Ordering::Greater` branch already answers a
too-high `ResendRequest` immediately (BUG-29's fix, line 825-827: `if mt.as_deref() == Some("2") {
actions.extend(self.on_resend_request(&msg)); }`), but the same message is *also* pushed onto
`self.queue` (line 828) unconditionally. When the session's own gap later closes and `drain_queue`
walks the queue in order, `process_in_order` (line 942-949) routes `Some("2")` back into
`on_resend_request` a **second time** — building and sending the entire resend/gap-fill response
again.

**QFJ**: `nextQueued(int)` (`Session.java:2473-2487`) explicitly special-cases `LOGON`/
`RESEND_REQUEST` in the drain loop: `if (MsgType.LOGON.equals(msgType) ||
MsgType.RESEND_REQUEST.equals(msgType)) { state.incrNextTargetMsgSeqNum(); }` — only the sequence
counter advances; the message itself is never reprocessed, since (per QFJ's own comment) "Logon and
ResendRequest processing has already been done."

**Impact**: Every too-high `ResendRequest` received while a gap is open causes a second, redundant
full resend/gap-fill burst once the gap closes — wasted bandwidth/processing at minimum, and (if the
store has since mutated, e.g. more messages arrived and were persisted) a resend response that
differs from the first one, confusing a counterparty that already processed the first response.

**Fix direction**: In `drain_queue`'s `process_in_order` dispatch, skip re-invoking
`on_resend_request` (and, by the same logic, `on_logon` if a queued Logon is ever possible) — only
advance `next_in_seq`, mirroring QFJ's `nextQueued(int)`.

### NEW-87: Latency/CompID/PossDup-falsification failures send only a `Logout`, never the `Reject` QFJ sends first

**Severity: P3 (wire-observable divergence, no functional break)**

**TF**: `crates/truefix-session/src/state.rs`'s `on_received` latency-failure branch (line 702-710)
and identity-problem branch (line 713-721) each send exactly one message: a `Logout`. Likewise
`poss_dup_falsification_check`'s caller (line 780-784) routes through `reject_logon` (Logout only).

**Fix direction**: For a non-Logon message, send a session Reject (correct
`SessionRejectReason`/`RefTagID`) before the Logout in the latency/CompID/PossDup-falsification

### NEW-90: MSSQL connections (both `truefix-store` and `truefix-log`) hardcode `trust_cert()` with no way to opt into real certificate validation

**Severity: P2 (security — silent, unconditional TLS validation bypass for any network-deployed MSSQL backend)**

**TF**: `crates/truefix-store/src/mssql.rs:102` (`parse_url`, path-form) and `:205`

```rust
config.authentication(AuthMethod::sql_server(
    percent_decode(user),
    percent_decode(password),

config.trust_cert();
```

`tiberius::Config::trust_cert()` unconditionally disables TLS server-certificate-chain validation.
There is no config field, URL parameter, or feature flag anywhere in either crate to opt back into

**QFJ/QFGo**: N/A directly (neither ships a first-party MSSQL/tiberius backend to compare against),
but this is a straightforward security-hygiene gap regardless of reference-implementation parity.

**Impact**: A MITM on the path between the FIX engine and its MSSQL sequence-number/message-log

full sequence-number state — a real confidentiality/integrity exposure for any non-localhost MSSQL

**Fix direction**: Add a config option (e.g. a `trust_server_certificate: bool` field, or a parsed

today's `trust_cert()` behavior for backward compatibility, but letting an operator disable it.

### NEW-91: Async DB-backed log writer tasks are never joined/flushed on `Engine::shutdown()` — queued log entries can be silently lost on shutdown

**Severity: P3 (robustness/edge-case — not reachable in normal operation, but a real gap under abrupt shutdown)**

**TF**: Each of `SqlLog`/`MssqlLog`/`MongoLog`/`RedbLog` spawns a background writer via a bare
`tokio::spawn(async move { while let Some(entry) = rx.recv().await { ... } })` (e.g.
`crates/truefix-log/src/sql.rs:311-327`, `mssql.rs:215-231`, `mongo.rs:93-121`, `redb.rs:137-171`)
and returns only an `mpsc::UnboundedSender<Entry>` — no `JoinHandle` is retained or exposed anywhere.
`crates/truefix/src/lib.rs`'s `Engine::shutdown`/`impl Drop for Engine` both call only
`abort_everything`, which aborts session tasks and has no reference to, and never awaits, any log
writer's background task (confirmed: neither function name appears anywhere in `truefix-log`).

**QFJ/QFGo**: N/A — `JdbcLog.java`'s writes are synchronous (no background task to lose); this is a
consequence of TrueFix's own fire-and-forget `tokio::spawn` design for async log backends, not a
parity gap.

**Impact**: If the process exits (or the tokio runtime is torn down) shortly after
`Engine::shutdown()`/`Drop`, any log entries still sitting in the unbounded channel that haven't yet
been written are lost — no error, no backpressure, no way for a caller to wait for drain.

**Fix direction**: Have each `connect_with_config` return (or store internally) the `JoinHandle`,
close the channel by dropping `tx` explicitly, and provide something a caller can await (e.g. an
async `flush()`/`shutdown()` on the `Log` trait, or make `Engine::shutdown` async and join
outstanding log-writer handles) before the process exits.

### NEW-92: `Engine::start`'s acceptor-group startup order is non-deterministic across process runs (`HashMap` iteration)

**Severity: P3 (does not produce a wrong result, only non-reproducible diagnostics on a partial-startup failure)**

**TF**: `crates/truefix/src/lib.rs:326` builds `acceptor_groups: HashMap<SocketAddr,
Vec<ResolvedSession>>`, then `:339` iterates it with `for (addr, members) in acceptor_groups`.
Rust's `HashMap` uses a per-process-random SipHash seed, so iteration order varies run to run for the
identical `.cfg`.

**Impact**: With `ContinueInitializationOnError=N`, the first group whose startup fails aborts
`Engine::start` and triggers `cleanup_partial_start` on whatever was already bound — which groups
count as "already bound" before the fatal one is hit varies from run to run for the same config. The
end state (full cleanup, `Err` returned) is always correct; only partial-startup diagnostics/logs
become non-reproducible. The non-grouped `singles` path (a `Vec`) is unaffected.

**Fix direction**: Iterate a stable, sorted order (e.g. sort by `SocketAddr` before the loop) so
partial-startup failure diagnostics are reproducible across runs of the same config.

### NEW-93: Multi-session acceptor's pre-Logon read has no timeout or buffer cap — unauthenticated memory-exhaustion / connection-hold DoS

**Severity: P1 (real, easily-reachable pre-auth DoS)**

**TF**: `crates/truefix-transport/src/lib.rs:1772-1798` (`route_and_run`, the multi-session
`AcceptorBuilder` path):

```rust
let mut buf: Vec<u8> = Vec::new();
let mut chunk = [0u8; 4096];
let logon = loop {
    match stream.read(&mut chunk).await {
        Ok(0) | Err(_) => return,
        Ok(n) => { if let Some(slice) = chunk.get(..n) { buf.extend_from_slice(slice); } }
    }
    match frame_length(&buf) {
        Ok(Some(total)) => match buf.get(..total).map(decode) { /* ... */ },
        Ok(None) => continue,
        Err(_) => return,
    }
};
```

`frame_length` (`crates/truefix-core/src/framing.rs:19-24`) returns `Ok(None)` immediately whenever
no SOH byte is present in `buf` yet — it never reaches the `MAX_BODY_LEN` check. A peer that opens a
connection and sends nothing, or sends bytes that never include a SOH (or trickles one byte every
few seconds), makes this loop `continue` forever: `buf` grows without any cap and the read has no
timeout — nothing here is bounded by `logon_timeout`, since that only starts once a `Session` object
exists, which happens *after* this loop already succeeded (confirmed by reading `run_connection`,
`lib.rs:731-865`: `Session::new` + the ticker that enforces `logon_timeout` both start only after
`route_and_run`'s loop returns a decoded Logon).

**Contrast with the single-session `Acceptor`**: `Acceptor::serve` → `spawn_session_io` →
`run_connection` directly, with the `Session`/ticker/`logon_timeout` live from the instant the
connection is accepted — a silent peer there is disconnected within `logon_timeout` (default 10s).
Only the multi-session (`AcceptorBuilder`) routing-by-Logon-content path is exposed.

**Impact**: An attacker can open many concurrent connections to a multi-session acceptor port and,
per connection, hold it open sending nothing (near-zero attacker cost) or trickle non-SOH garbage —
each consumes a live task + open socket indefinitely, and any connection that does send data
accumulates it in `buf` uncapped. Classic slowloris-shaped pre-auth DoS.

**Fix direction**: Wrap this loop's `stream.read()` in a `tokio::time::timeout` (reuse
`logon_timeout` or add a dedicated setting), and/or cap `buf`'s size defensively regardless of
whether a SOH has appeared yet.

### NEW-94: PROXY protocol `Unknown`/`Unspecified`/Unix-socket headers are fully parsed but their byte-length is discarded — header bytes never consumed, misread as FIX data

**Severity: P2 (real, but narrow — only affects trusted-proxy deployments emitting a spec-legal `PROXY UNKNOWN` v1 header or a v2 header with an unspecified/Unix address family)**

**TF**: `crates/truefix-transport/src/proxy.rs:156-186`:

```rust
HeaderResult::V1(Ok(header)) => {
    let len = header.header.len();
    let addr = match header.addresses {
        v1::Addresses::Tcp4(a) => ...,
        v1::Addresses::Tcp6(a) => ...,
        v1::Addresses::Unknown => return None,   // `len` is known but discarded
    };
    Some((addr, len))
}
HeaderResult::V2(Ok(header)) => {
    let len = header.len();
    let addr = match header.addresses {
        v2::Addresses::IPv4(a) => ...,
        v2::Addresses::IPv6(a) => ...,
        v2::Addresses::Unspecified | v2::Addresses::Unix(_) => return None, // same
    };
    Some((addr, len))
}
```

Both branches compute `len` (the correctly-parsed header's byte length) before matching on address
type, but the `Unknown`/`Unspecified`/`Unix(_)` arms (all spec-legal PROXY protocol values — `PROXY
UNKNOWN` is standard for e.g. health-check connections) discard it and return `None`. The caller
(`peek_proxy_header`) treats `None` identically to "not a complete header yet," keeps re-peeking,
then eventually gives up and returns the physical peer address **without ever consuming the header's
bytes** — they're left in the stream and misread as the start of the FIX message.

**Distinction from NEW-12**: NEW-12 is about a header too large for the peek buffer (truncated/
oversized input). NEW-94 is a *fully and correctly parsed, spec-legal* header being mishandled after
the fact — a different root cause with the same symptom.

**Fix direction**: Return `Some((None, len))` (or a 3-way enum) instead of `None` for these branches,
so the caller can still consume+discard the correct number of header bytes even when it declines to
remap the address.

### NEW-95: TLS trust-store loading silently drops every malformed certificate — a fully-broken trust-store file produces an empty, non-functional `RootCertStore` with zero diagnostic

**Severity: P3 (real diagnostic gap; narrower trigger than NEW-11, which is about no trust store being configured at all)**

**TF**: `crates/truefix-transport/src/tls_config.rs:117-135` (`load_root_store` and

```rust
let mut roots = RootCertStore::empty();
for cert in load_certs(path)? {

```

Every certificate that fails `RootCertStore::add` (malformed DER, unsupported key type, etc.) is

**Fix direction**: Track how many certs were successfully added; if zero (or if any failed), surface
it — at minimum a log warning with the failure count, or a typed error when the resulting store is
empty despite non-empty input.

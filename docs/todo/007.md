## NEW-156: `FileLog` performs synchronous blocking I/O directly on the tokio executor

**Category:** Bug / Perf-Hygiene

**Severity:** P2

**TF**: `crates/truefix-log/src/file.rs` (`RotatingFile::write_line`, unconditional `writeln!`);
called from `crates/truefix-transport/src/lib.rs` inside `tokio::spawn`'d async fns (`read_loop`,
`dispatch`, `handle_inbound`, ~L723/L922/L1320/L1440/L1517/L1528).

`FileLog::on_incoming`/`on_outgoing`/`on_event` write to `messages.log`/`event.log` synchronously
(`std::fs::File` + `writeln!`) and are called directly from async functions running on tokio
worker threads — there is no `spawn_blocking` or background-writer hop. This is inconsistent with
the discipline already established elsewhere in the same crate: `RedbLog`
(`crates/truefix-log/src/redb.rs`) explicitly wraps its synchronous `redb` calls in
`tokio::task::spawn_blocking` "to never block the async executor" (mirroring `RedbStore`'s
convention in `truefix-store`), and `SqlLog`/`MssqlLog` push entries onto a bounded `mpsc` channel
consumed by a background task rather than writing inline.

`ScreenLog` and `TracingLog` have the same property (synchronous, called inline) but are lower
risk: stdout/stderr and the `tracing` facade are typically non-blocking or buffered by the process
supervisor, whereas `FileLog` does a real `write(2)` syscall per line, with no buffering beyond the
OS page cache.

**Impact**: Under normal conditions (local disk, page-cache-backed writes) the per-call cost is
negligible. Under disk contention, a slow/network-backed log volume, or an `fsync`-heavy
environment, `writeln!` can block the calling tokio worker thread for the duration of the syscall,
stalling every other task scheduled on that thread — including other sessions' heartbeat timers
and message processing on a multi-threaded runtime with few worker threads. This is a tail-latency
and (in the worst case) fairness risk, not a correctness bug.

**Fix direction**: Reuse the pattern already in this crate rather than inventing a new one — either
(a) route `FileLog` writes through the same bounded-channel + background-task shape as
`SqlLog`/`MssqlLog`/`MongoLog`/`RedbLog` (giving it a real `shutdown()` override to flush on
drain, instead of the default no-op), or (b) wrap each `write_line` call in
`tokio::task::spawn_blocking`, matching `RedbLog`'s simpler approach. (a) is preferable for
consistency with the rotation/backpressure semantics of the other backends and for keeping
`on_*`'s current fire-and-forget, infallible-to-the-caller contract.

## NEW-157: `FileLog` has no multi-generation or time-based rotation policy

**Category:** Feature-Gap

**Severity:** P3

**TF**: `crates/truefix-log/src/file.rs` `RotatingFile::rotate()` — renames the current file to a
single fixed `<name>.1`, overwriting any previous backup; no generation count, no compression, no
date-based file naming.

**QFJ**: QuickFIX/J's own `FileLog` has no rotation at all (unbounded growth by default). However,
QFJ ships an `SLF4JLogFactory` (`quickfix.SLF4JLogFactory`/`SLF4JLog`) as an alternative to
`FileLog`, which defers all log output to an SLF4J binding — in production QFJ deployments this is
almost always Logback, configured with a `RollingFileAppender` +
`TimeBasedRollingPolicy`/`SizeAndTimeBasedRollingPolicy` (daily/hourly file naming, N-generation
retention, optional gzip of rolled files). So the *capability* the user wants — proper
multi-generation and/or time-sliced rotation — has real precedent in the QFJ ecosystem, just
delivered by the logging framework rather than QFJ's own `FileLog`.

TrueFix has no `tracing`-subscriber-based equivalent path wired up for the message/event log
streams today: `TracingLog` (`crates/truefix-log/src/tracing_log.rs`) emits through the `tracing`
facade, but nothing in this repo configures a rolling `tracing-appender` (or similar) as the
sink for it, and `FileLog` is a hand-rolled writer independent of `tracing` entirely.

**Impact**: Operators who need bounded, auditable log retention (e.g. "keep 30 daily files,
compress after 1 day") currently have to bolt on external tooling (`logrotate` with
`copytruncate`, since `FileLog` holds a long-lived file handle and won't notice an external
rename) or accept the single-backup `MaxFileLogSize` behavior from `NEW-108`, which is
intentionally minimal.

**Fix direction**: Two viable paths, not mutually exclusive:
1. Extend `RotatingFile`/`FileLogOptions` with a real generation-count policy (`<name>.1` ..
   `<name>.N`, shifting on rotate instead of overwriting) and/or a `TimeBasedRollingPolicy`-style
   date-stamped file naming (roll at UTC-midnight or a configurable interval), matching the
   options Logback exposes.
2. Add a `tracing-appender`-backed log sink (`tracing_appender::rolling`) as another `LogConfig`
   variant, giving TrueFix users the same "day/hour rolling, non-blocking writer" combination that
   QFJ users get for free via `SLF4JLogFactory` + Logback — this would also naturally fold in
   NEW-156's async-writer fix, since `tracing-appender`'s `non_blocking` wrapper already does the
   channel + background-thread hop.

Given the size of this change (new config surface: retention count, roll interval, compression),
this should be scoped as its own feature rather than folded into a quick fix, and should land
together with NEW-156 so the new rotation path is async from the start rather than needing a
second migration later.

## NEW-158: `Engine::start`/`SessionSettings` (`.cfg`) path cannot accept a user-supplied `MessageStore`/`Log` backend

**Category:** Feature-Gap

**Severity:** P3

**TF**: `crates/truefix/src/lib.rs:335` (`Engine::start`) takes only `&SessionSettings`, resolved
internally into `truefix_config::SessionConfig` (`crates/truefix-config/src/builder.rs:67`) whose
`store: StoreConfig` and `log`/`log_kind`/`sql_log` fields are closed enums/structs — they only
name the backends already built into the crate (Memory/File/CachedFile/Noop/Sql/Mssql/Redb/Mongo
for store; Screen/File/Tracing/Composite/Sql/Mssql for log). `Engine::start` resolves these via
`truefix_store::build_store(&primary.store)` (~L482) and `build_log`/`build_log_kind` (~L486/L490).
Neither `Engine::start` nor any `SessionConfig` field accepts a pre-built `Box<dyn MessageStore>`/
`Box<dyn Log>` — a third party cannot inject a custom backend through this path.

This is despite both extension points already being open at the trait level: `MessageStore`
(`crates/truefix-store/src/lib.rs:69`, `pub trait MessageStore: Send + Sync`, `#[async_trait]`,
all-`pub`-typed methods, no sealed supertrait) and `Log` (`crates/truefix-log/src/lib.rs:76`, same
shape, plus a convenience `impl Log for Box<dyn Log>` at L91-105) are both fully public and
externally implementable — `build_store`/`build_log` themselves return `Box<dyn MessageStore>`/
`Box<dyn Log>` (proving object-safety), so nothing about the traits blocks a third-party impl.

**What already works today**: the lower-level `truefix-transport` API — fully re-exported as
`truefix::transport` (`crates/truefix/src/lib.rs:32-34`) — *does* accept user-supplied backends
directly. `Services` (`crates/truefix-transport/src/lib.rs:250-289`) has public
`store: Option<Arc<dyn MessageStore>>` (L254) and `log: Option<Arc<dyn Log>>` (L256) fields,
accepted by `Acceptor::bind_with(..., services: Services)` (L649),
`AcceptorBuilder::with_services`/`with_session_store` (L1929/L1858), and
`connect_initiator_with`/`connect_initiator_tls`/`connect_initiator_via_proxy` (L502/L518/L542).
So extensibility already exists end-to-end — it's just unreachable from the `.cfg`-driven
`Engine::start` convenience API, which is the primary/documented entry point most users will reach
for first.

**Impact**: Low-severity ergonomics gap, not a hard blocker (the `transport` escape hatch exists
today and is usable). But it's a discoverability trap: a user who wants a custom store/log (e.g. a
proprietary audit-log sink, or a store backend not on the built-in list) and starts from
`SessionSettings`/`Engine::start` — the path shown in the README/quickstart — has no way to
discover that dropping down to `truefix::transport::{Acceptor, Services}` is necessary, since
`StoreConfig`/`LogConfig` give no indication that anything sits outside their variants.

**Fix direction**: Two options, not mutually exclusive:
1. Add a `Custom(Arc<dyn MessageStore>)`/`Custom(Arc<dyn Log>)` variant to `StoreConfig`/`LogConfig`
   (or an equivalent `LogKind`/`SqlLogSpec`-adjacent variant), so `.cfg`-driven config and
   programmatic construction can both express "use this externally-provided backend." Requires
   these enums to stop being pure data (`Debug, Clone`) if the variant holds a trait object —
   check downstream `Clone`/`Debug` bound usage before committing to this shape.
2. Give `Engine`/`SessionConfig`'s builder an explicit override hook (e.g.
   `SessionConfig::with_store(Arc<dyn MessageStore>)`/`with_log(Arc<dyn Log>)`, or an
   `Engine::start_with_services(&SessionSettings, Services)` overload) that layers a caller-supplied
   `Services` on top of the `.cfg`-resolved one, reusing the mechanism `truefix-transport` already
   has rather than duplicating it in `truefix-config`.

Should be documented (README/quickstart) regardless of which fix lands, since the escape hatch via
`truefix::transport` already works and just needs to be surfaced.

## Summary

| ID | Category | Severity | Summary |
|----|----------|----------|---------|
| NEW-156 | Bug / Perf-Hygiene | P2 | `FileLog` blocks the tokio executor with inline synchronous file I/O |
| NEW-157 | Feature-Gap | P3 | No multi-generation or time-based rotation (Logback-style) for `FileLog` |
| NEW-158 | Feature-Gap | P3 | `Engine::start`/`.cfg` path can't accept a user-supplied `MessageStore`/`Log`; only the lower-level `transport::Services` path can |

### Suggested fix priority

1. **NEW-156** — cheaper, fixes a real (if usually latent) tail-latency/fairness risk; do this
   first since NEW-157's rotation rewrite should be built on the async path, not migrated onto it
   later.
2. **NEW-157** — larger scope (new config surface), land together with or immediately after
   NEW-156. Decide between "extend `RotatingFile` directly" vs. "add a `tracing-appender`-backed
   sink" before starting; the latter buys more (async + battle-tested rolling policy) for
   comparable effort but changes `truefix-log`'s architecture more (a `tracing`-facade path for
   file logs, alongside the existing hand-rolled one).
3. **NEW-158** — independent of NEW-156/157 (it's about the *construction* API, not `FileLog`
   internals); cheapest fix is documentation (point users at `truefix::transport::Services` today),
   with the `Custom(...)` enum variant or `with_store`/`with_log` builder hook as the real fix when
   there's bandwidth. Low urgency since a working escape hatch already exists.
